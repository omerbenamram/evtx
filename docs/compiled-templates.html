The shape of the problem

An EVTX file is a sequence of 64 KiB chunks, each holding a few hundred records. A record's payload is binary XML (BinXML): a tokenized tree format in which Microsoft made one excellent decision — they separated structure from data. The structure lives in a template definition (an element tree with typed holes), stored once per chunk. Each record is then just a template instance: a reference to a definition plus an array of typed substitution values.

Rendered, a record looks like this — a Security 4688, process creation:

<Event xmlns="http://schemas.microsoft.com/win/2004/08/events/event">
  <System>
    <Provider Name="Microsoft-Windows-Security-Auditing" Guid="{54849625-…}"/>
    <EventID>4688</EventID>
    …
    <TimeCreated SystemTime="2016-10-06T01:47:07.166302Z"/>
    <Computer>WIN-WFBHIBE5GXZ.example.co.jp</Computer>
  </System>
  <EventData>
    <Data Name="NewProcessName">C:\Windows\System32\cmd.exe</Data>
    …
  </EventData>
</Event>

Almost every byte of that output is constant per template. Only the values change between records. And the workload is brutally repetitive: our 30 MB benchmark log holds 62,000 records across 481 chunks but only 32 distinct templates. Before this round, the parser re-walked a template's structure ~4,500 times per file — once per chunk per template, and then again, in spirit, for every single record.

Where three generations spent their time

The profile of gen 2 made the bet obvious. Roughly a quarter of single-thread time decoded values into an intermediate enum, another sixth dispatched the emitter over the tree, a sixth escaped and wrote strings, and a seventh ran render-time layout scans — per record, for structure that is identical across thousands of records. The only honest fix was to stop re-deriving and start compiling.

The compiler

For each (template definition, output format) pair we now walk the definition once and partially evaluate everything static into a flat splice program: a byte buffer of pre-rendered literal output (already escaped, already indented, already serialized for the target format) interleaved with ops that say where record values get spliced in.

The XML op set is five variants; everything an op needs to make a runtime decision (is this value empty? is it a nested element?) is precomputed into it, including the alternative byte tails:

enum XOp {
    /// Splice lits[range] into the output.
    Lit(LitRange),
    /// Escaped value text (element content or always-present attribute).
    Val { slot: u16, in_attr: bool },
    /// ` name="value"` — the whole attribute vanishes when the value is empty.
    AttrVal { slot: u16, pre: LitRange },
    /// Element content with one hole: empty / text / nested-element each
    /// get a precompiled closing tail.
    Body { slot: u16, optional: bool, indent: u16,
           tail_text: LitRange, tail_empty: LitRange, tail_elem: LitRange },
    /// A hole in child position: nothing, nested instance, or indented text.
    ChildSlot { slot: u16, optional: bool, indent: u16, ind: LitRange },
}

JSON compiles against the same IR with its own four-op set (Lit, LeafVal, Elem, SlotChild) that bakes in the trickier JSON semantics: EventData/UserData flattening, duplicate-key _N suffixes counted at compile time, the number-coercion rule (only integers and booleans go bare; floats, hex, timestamps, GUIDs and SIDs stay quoted), and null-vs-"" empty forms.

Per record: scan, then execute

At render time a record is one header read plus its substitution descriptor table — n entries of (size, type). The pre-flight turns that into a table of raw slots, each a typed window into the chunk buffer:

struct RawSlot {
    off: u32,      // value's offset in the chunk
    len: u16,      // declared size
    ty: u8,        // wire type
    nested: u16,   // resolved nested-instance program, if any
    ansi: u32,     // pre-decoded ANSI string, if any
}

Validation is driven by a 256-entry const facts table built from the wire spec — the declarative replacement for a per-type match cascade (and measurably faster):

const TY_CLASS: [TyClass; 256] = { /* NULL→Sized, BOOL→Fixed(4),
    GUID→Fixed(16), SID→Sid, UTF16_STRING→Utf16, …, everything else→Reject */ };

The executor is then a linear loop over the ops, formatting each value once, directly from chunk bytes into the output buffer — UTF-16 → UTF-8 with escaping fused in a SIMD path, integers via fixed-width decoders, no intermediate allocation. Values never materialize into an enum at all.

One walker, two lanes — no fallback renderer

The dangerous way to build this is a fast path with its own rendering logic and a "fallback" for everything else: two implementations of the same semantics, drifting apart one edge case at a time. That was rejected outright. The rule for this codebase is:

A bail is lane selection, not a second renderer. The compiler is the renderer: walking a template it records ops at the holes; walking a materialized tree (where every hole is already resolved) the same code renders directly to the output. One implementation, two modes, identical decisions at every site.

This is also where the campaign removed code: the unification stages deleted the standalone XML/JSON emitters and the render-time resolution layer — about 1,500 net lines — while keeping the speed. Wire-format facts got names along the way; the parser now reads like the spec it implements:

/// Offset of the TemplateInstance body in a (possibly fragment-wrapped)
/// single-instance stream; None for any other shape.
const fn single_instance_offset(bytes: &[u8]) -> Option<usize> {
    match bytes {
        [token::FRAGMENT_HEADER, _, _, _, token::TEMPLATE_INSTANCE, _, ..] => Some(5),
        [token::TEMPLATE_INSTANCE, _, _, _, _, _, ..] => Some(1),
        _ => None,
    }
}

Compile once per file, not per chunk

Programs own their bytes (no lifetime ties to a chunk), so they are cached at two levels: a per-chunk map keyed by definition offset for the common case, backed by a parser-wide store keyed by template content identity — (GUID, size, hash of definition bytes) — behind an RwLock, shared by all worker threads via Arc. The 481 chunks of the benchmark file used to trigger ~4,500 template parses per read; now each of the 32 templates compiles exactly once per run, and multithreaded workers reuse each other's work.

Keeping it honest

  • Byte-identity: 27 sample logs (every odd shape in the corpus: ANSI codepages, WEVT-cached templates, separate-attribute JSON, zero-record files) × 5 output modes, diffed against the previous release after every commit. The sweep caught one real bug during development — generic BinXML fragments in JSON — before it ever landed.
  • Tests: 99 integration tests plus insta snapshots, which also gate the per-record APIs.
  • Adversarial review: a correctness pass over the new path found (and fixed) one genuine pre-existing DoS — unbounded recursion on crafted nesting chains, now capped at build, validation, and scan time — and the program store now recovers from lock poisoning instead of cascading panics.
  • #![forbid(unsafe_code)] — all of this is safe Rust; the SIMD UTF-16 path lives behind the same guarantee.

What's left on the table

Arrays. A substitution can be array-typed, repeating its containing element once per item — System and Application logs have arrays in 17–23% of records, and those currently take the slow lane. Teaching the op set to expand arrays (Expand ops) is the last big structural bet. After that: a compiled variant for separate-attributes JSON, and short-string escape tails in the SIMD writer.

Benchmarks: hyperfine -w2 -r10, evtx_dump -t 1 -o <fmt> -f /dev/null, fast-alloc feature, Apple M3 Pro (macOS) and AMD Zen 2 (Linux, taskset-pinned). Output hashes and the full mode sweep are reproducible from the repo's samples directory.
